## Push / deploy / revert (RECOMMENDED)

### Before first push

1. Run `alembic upgrade head` on prod **once** during deploy.
2. Keep `partner_enabled: false` until a store is onboarded.
3. Push migration files **with** the code.

### If things go wrong after push

**Recommended: revert CODE only — do NOT roll back DB.**

```bash
git checkout origin/main
# or redeploy previous build
```

- Old app **ignores** extra table + nullable columns — WhatsApp/dashboard keep working.
- Set `partner_enabled: false` in settings to disable partner without schema change.

**Do not** run `alembic downgrade` on prod unless you have a maintenance window and know the revision chain.

### Optional: clean local dev DB only

```sql
DROP TABLE IF EXISTS partner_webhook_deliveries;
ALTER TABLE orders DROP COLUMN IF EXISTS pos_push_status;
ALTER TABLE orders DROP COLUMN IF EXISTS pos_pushed_at;
ALTER TABLE orders DROP COLUMN IF EXISTS pos_order_id;
DROP INDEX IF EXISTS ix_orders_restaurant_status_created;
```

Do **not** drop `partner_api_keys` — used by pre-existing partner key feature on main.

### Code revert vs DB (remember)

| Action | Reverts code | Reverts DB |
|--------|--------------|------------|
| `git checkout` / `git reset` | Yes | **No** |
| `alembic downgrade` | No | Yes |
| `partner_enabled: false` | No | No (disables feature only) |

## Webhooks vs APIs

| Direction | Mechanism | Use for |
|-----------|-----------|---------|
| **Outbound webhook** | We → POS `POST` | order.created, rider_assigned, picked_up, delivered, late |
| **Inbound API** | POS → us `X-API-Key` | kitchen status, menu, ack, poll backup |

- **Kitchen = always API** (`POST .../status`) — triggers dispatch on `ready`.
- **Orders + delivery = webhooks** (primary) + **poll APIs** (backup).

### Auth

| Who | How |
|-----|-----|
| POS → us | `X-API-Key: rk_live_...` (60/min per key, `APP_PARTNER_RATE_LIMIT`) |
| Us → POS | HMAC `X-Partner-Signature: sha256=...` |
| Manager | JWT Bearer |

Revoked key → 401. Authed partner calls → `audit_log` `actor=pos`.

### Outbound webhook headers

```
Content-Type: application/json
X-Partner-Event: order.created
X-Partner-Idempotency-Key: pos-order-created-42
X-Partner-Signature: sha256=<hmac_hex>
```

**Envelope:** `{ "event", "idempotency_key", "timestamp", "data" }`  
**Events:** `integration.ping` · `order.created` · `order.rider_assigned` · `order.picked_up` · `order.delivered` · `order.late`  
**Retry:** 5× backoff (10s, 20s, 40s, 80s) → dead. POS returns 2xx within 5s.  
**Doc:** `docs/partners/webhook-events.md`

### Inbound APIs (`/api/v1/partner/...`)

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/store` | Store + integration flags |
| GET | `/orders` | Poll orders (`?unacked_only=true`) |
| GET | `/orders/{id}` | One order |
| POST | `/orders/{id}/ack` | `{ "pos_order_id": "..." }` |
| POST | `/orders/{id}/status` | `preparing` / `ready` / `cancelled` |
| GET | `/orders/{id}/delivery` | Rider, ETA, COD poll |
| GET | `/riders/{id}/location` | GPS (1 req/10s) |
| PUT | `/menu/items` | Bulk menu upsert |
| PATCH | `/menu/items/{pos_id}` | Price / sold-out |
| POST | `/events/menu-changed` | Queue full POS pull |
| GET | `/menu/sync-status` | Sync breadcrumb |
| GET | `/customers` | Incremental customer pull |

### Manager JWT only

| Method | Endpoint |
|--------|----------|
| GET/PATCH | `/partner-integration/config` |
| POST | `/partner-integration/webhooks/test` |
| GET | `/partner-integration/health` |
| POST/GET/DELETE | `/api/v1/api-keys` |

**OpenAPI:** `docs/partners/openapi-partner-v1.yaml`  
**Postman:** `docs/partners/postman-partner-v1.json`

### Per-store config (`restaurants.settings`)

```json
{
  "partner_enabled": true,
  "partner_webhook_url": "https://pos.example.com/hooks/whatsapp",
  "partner_webhook_secret": "shared-secret",
  "pos_store_id": "CRT-DXB-014",
  "pos_order_push_mode": "webhook"
}
```

Poll mode: empty `partner_webhook_url` + `pos_order_push_mode: poll` → POS uses `GET /partner/orders`.

## Critical path (one order)

```
1. Customer confirms on WhatsApp
2. order.created webhook → POS                    [Phase 1]
3. POS POST .../ack                                 [Phase 1]
4. POS POST .../status preparing                    [Phase 2]
5. POS POST .../status ready → dispatch → rider    [Phase 2 — MAIN ENGINE]
6. order.rider_assigned webhook → POS               [Phase 4]
7. Rider pickup → order.picked_up → POS           [Phase 4]
8. Rider delivered → order.delivered + COD → POS  [Phase 4]
9. (optional) order.late + coupon → POS            [Phase 4]
```

## Webhook pipeline (technical)

```
event → enqueue_partner_webhook() [same transaction]
     → commit
     → flush_pending_partner_webhooks()
     → Celery partner.deliver_webhook (or sync if APP_OUTBOX_SYNC_DELIVERY=true)
     → POST partner_webhook_url [HMAC]
     → 2xx=sent | retry 5x | dead
```

| Event | Enqueued from |
|-------|---------------|
| `order.created` | `ordering/service.py → finalize_confirmation()` |
| `order.rider_assigned` | `dispatch/service.py → _commit_route()` |
| `order.picked_up` / `delivered` | `dispatch/delivery.py → advance_delivery()` |
| `order.late` | `sla/monitor.py → _fire_event(breach_40)` |

Kitchen inbound only:

```
POST /api/v1/partner/orders/{id}/status {"status":"ready"}
→ apply_partner_kitchen_status() → advance_kitchen_status() → _auto_dispatch_on_ready()
```

## Architecture

**Single source of truth:** our DB. POS API, webhooks, and dashboard all use the same domain services.

```mermaid
flowchart TB
    WA[Customer WhatsApp] --> BOT[Conversation Engine]
    BOT --> API[Partner API + Webhooks]
    POS[Partner POS] -->|menu + kitchen API| API
    API -->|order + delivery webhooks| POS
    API --> DISPATCH[Dispatch + SLA]
    DISPATCH --> RIDER[Rider App]
    DISPATCH --> WA
    API --> META[Meta Catalog]
    META --> WA
    DASH[Dashboard optional] --> API
```

| Actor | UI | Role |
|-------|-----|------|
| Customer | WhatsApp | Order, track |
| POS | Their screens | Menu, kitchen, billing |
| Platform | APIs + workers | Bot, dispatch, SLA, rider app |
| Rider | Android app | Pickup, deliver, COD |
| Restaurant | POS required; dashboard optional | Daily ops |
