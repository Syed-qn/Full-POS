## Golden Rules (read before every phase)

1. **One brain** ‚Äî POS API, dashboard, and WhatsApp all call the SAME service functions:
   - Kitchen: `ordering/service.py ‚Üí advance_kitchen_status()`
   - Dispatch: already fires inside `advance_kitchen_status()` when status = `ready`
   - Menu: `pos/sync_service.py ‚Üí sync_menu_from_pos()`
   - Cancel: `ordering/service.py ‚Üí cancel_order()`

2. **Thin routers** ‚Äî `partner/router.py` only validates input, calls service, returns JSON.

3. **New module layout** (add gradually, do NOT dump everything in router.py):
   ```
   src/app/partner/
     router.py          ‚Üê HTTP only (already exists)
     schemas.py         ‚Üê request/response models
     deps.py            ‚Üê X-API-Key auth (already exists)
     service.py         ‚Üê NEW: shared partner helpers
     orders_api.py      ‚Üê NEW: order list/ack/status handlers
     menu_api.py        ‚Üê NEW: menu push handlers (Phase 3)
     webhooks/
       models.py        ‚Üê partner_webhook_deliveries table
       enqueue.py       ‚Üê write outbound webhook row in same DB transaction
       deliver.py       ‚Üê HTTP POST to POS with HMAC + retry
       signing.py       ‚Üê HMAC-SHA256 sign/verify
   ```

4. **Outbound webhooks** = separate table from WhatsApp outbox.
   WhatsApp outbox has `to_phone`. POS webhooks have `target_url` + JSON body.
   Same pattern: write in transaction ‚Üí worker delivers with retry.

5. **Inbound from POS** = normal REST endpoints (`X-API-Key`), NOT us receiving
   their webhook unless they insist. Simpler: POS calls our API directly.

## How to Create Webhooks (Detailed Guide)

You already have the pattern ‚Äî WhatsApp **outbox**. POS webhooks are the same idea:
instead of sending to a phone number, you POST JSON to the POS URL.

### Two directions (do not mix them up)

| Direction | Who calls whom | How |
|-----------|----------------|-----|
| **Outbound webhook** (we ‚Üí POS) | We notify POS when something happens | Queue row ‚Üí worker POSTs to their URL |
| **Inbound API** (POS ‚Üí us) | POS tells us something happened | Normal REST endpoint + `X-API-Key` |

**Recommendation:** Use **outbound webhooks** for orders and delivery updates.
Use **inbound API** for kitchen status (`ready`, `preparing`) ‚Äî simpler and more reliable
than POS sending webhooks to us.

### Outbound webhook ‚Äî 4 pieces

```
Business event (order confirmed)
        ‚Üì
enqueue_partner_webhook()     ‚Üê write row in SAME transaction
        ‚Üì
DB commit
        ‚Üì
Celery worker picks up row
        ‚Üì
POST https://pos.com/hooks/orders  (HMAC signed)
        ‚Üì
POS returns 200 ‚Üí mark sent
POS fails       ‚Üí retry with backoff ‚Üí dead after 5 tries
```

Parallel in codebase today:

```
enqueue_message() ‚Üí outbox_messages ‚Üí deliver_outbox_message task ‚Üí Meta API
```

Copy retry logic from `src/app/outbox/worker.py` (`_backoff_countdown`,
`_is_permanent_failure`, claim pattern).

### Step 1 ‚Äî Store POS webhook URL per restaurant

In `restaurants.settings` JSONB (no migration):

```json
{
  "partner_enabled": true,
  "partner_webhook_url": "https://pos.example.com/hooks/whatsapp",
  "partner_webhook_secret": "shared-secret-for-hmac",
  "pos_store_id": "CRT-DXB-014",
  "pos_order_push_mode": "webhook"
}
```

POS gives you their URL during onboarding. Store once per store.

### Step 2 ‚Äî Delivery queue table

New table `partner_webhook_deliveries` (separate from WhatsApp `outbox_messages`):

| Column | Purpose |
|--------|---------|
| `restaurant_id` | Which store |
| `event_type` | e.g. `order.created` |
| `payload` | JSON body to send |
| `target_url` | Copied from settings at enqueue time |
| `idempotency_key` | Unique ‚Äî prevents duplicate sends on retry |
| `status` | `pending` ‚Üí `sent` / `dead` |
| `attempts` | Retry count |
| `last_error` | Last failure message |
| `delivered_at` | When POS returned 2xx |

### Step 3 ‚Äî Enqueue function (call from business code)

```python
# partner/webhooks/enqueue.py

async def enqueue_partner_webhook(
    session,
    *,
    restaurant_id: int,
    event_type: str,          # "order.created"
    payload: dict,
    idempotency_key: str,     # "pos-order-created-42"
) -> None:
    settings = await get_restaurant_settings(session, restaurant_id)
    url = settings.get("partner_webhook_url")
    if not url:
        return  # POS not configured ‚Äî skip silently

    row = PartnerWebhookDelivery(
        restaurant_id=restaurant_id,
        event_type=event_type,
        payload=payload,
        target_url=url,
        idempotency_key=idempotency_key,
        status="pending",
    )
    session.add(row)
    # caller commits ‚Äî same transaction as order confirm
```

**Hook point** ‚Äî after order becomes `confirmed` in `ordering/service.py`:

```python
await enqueue_partner_webhook(
    session,
    restaurant_id=order.restaurant_id,
    event_type="order.created",
    payload=build_order_payload(order),
    idempotency_key=f"pos-order-created-{order.id}",
)
await session.commit()
# then trigger Celery worker for the new row
```

### Step 4 ‚Äî HMAC signing

```python
# partner/webhooks/signing.py
import hmac, hashlib

def sign_body(secret: str, body: bytes) -> str:
    sig = hmac.new(secret.encode(), body, hashlib.sha256).hexdigest()
    return f"sha256={sig}"
```

Delivery headers:

```
Content-Type: application/json
X-Partner-Signature: sha256=<hmac of raw body>
X-Partner-Event: order.created
X-Partner-Idempotency-Key: pos-order-created-42
```

POS verifies signature with the same shared secret.

### Step 5 ‚Äî Celery worker

```python
# partner/webhooks/worker.py

@shared_task(bind=True, max_retries=5)
def deliver_partner_webhook_task(self, delivery_id: int):
    # 1. Load row (status must be pending)
    # 2. POST to target_url with HMAC headers
    # 3. 2xx ‚Üí status = sent
    # 4. 5xx/timeout ‚Üí retry with backoff (10s, 20s, 40s, 80s, 160s)
    # 5. 4xx (except 429) ‚Üí status = dead (permanent fail)
```

After commit:

```python
deliver_partner_webhook_task.apply_async(args=[row.id], queue="maintenance")
```

### Webhook events we send (outbound)

| Event | When we fire it | POS uses it for |
|-------|-----------------|-----------------|
| `order.created` | Customer confirms on WhatsApp | Show order in kitchen screen |
| `order.modified` | Customer changes order pre-ready | Update kitchen ticket |
| `order.cancelled` | Either side cancels | Remove from kitchen |
| `order.rider_assigned` | Dispatch assigns rider | Show rider on the way |
| `order.picked_up` | Rider taps pickup | Show out for delivery |
| `order.delivered` | Rider taps delivered | Close order, record COD |
| `order.late` | SLA breach | Flag in POS |

**Example `order.created` payload:**

```json
{
  "event": "order.created",
  "idempotency_key": "pos-order-created-42",
  "timestamp": "2026-07-01T13:37:00Z",
  "data": {
    "order_id": 42,
    "order_number": "R1-0042",
    "pos_store_id": "CRT-DXB-014",
    "customer": { "name": "Asfer", "phone": "+9715..." },
    "items": [
      { "dish_number": 110, "name": "Grill Mandi", "qty": 5, "price": 40.00 }
    ],
    "address": {
      "building": "Tower B", "room": "123", "lat": 25.1, "lng": 55.2
    },
    "subtotal": 200.00,
    "delivery_fee": 10.00,
    "total": 210.00,
    "payment": "COD",
    "promised_eta_minutes": 40
  }
}
```

### What POS must build (their side)

One endpoint receives all events:

```
POST https://their-pos.com/hooks/whatsapp-orders
```

Their handler must:

1. Verify `X-Partner-Signature` HMAC
2. Dedupe on `idempotency_key` (return 200 if already processed)
3. Route by `event` type
4. Return `200` within 5 seconds

**Fallback if POS cannot receive webhooks:** they poll
`GET /api/v1/partner/orders` instead (Phase 1).

### Inbound: POS calls our API (not a webhook)

Kitchen status ‚Äî POS calls us directly:

```
POST https://your-api.com/api/v1/partner/orders/42/status
Header: X-API-Key: <their key>
Body: { "status": "ready" }
```

‚Üí calls `advance_kitchen_status(actor="pos")` ‚Üí dispatch fires automatically.

No inbound webhook receiver needed for kitchen ‚Äî simpler auth, direct OPS engine hook.

### Webhook mental model

```
OUTBOUND (webhooks we create):
  order confirmed  ‚Üí enqueue ‚Üí worker POST ‚Üí POS kitchen screen
  rider delivered  ‚Üí enqueue ‚Üí worker POST ‚Üí POS order closed

INBOUND (API POS calls):
  POS "ready"      ‚Üí POST /partner/orders/{id}/status ‚Üí advance_kitchen_status()
  POS menu update  ‚Üí PUT /partner/menu/items ‚Üí sync_menu_from_pos()
```

### Webhook anti-patterns (do NOT)

- Do NOT POST to POS synchronously inside the WhatsApp webhook handler ‚Äî always queue
- Do NOT put POS deliveries in `outbox_messages` ‚Äî that table is for phone numbers only
- Do NOT call POS before DB commit ‚Äî enqueue in same transaction, deliver after commit
- Do NOT skip `idempotency_key` ‚Äî retries will duplicate orders in POS without it

## Phase 1 ‚Äî Orders OUT to POS (the intake loop, ~1 week)
**Goal:** WhatsApp order confirmed ‚Üí POS receives it. OPS can see the order in POS kitchen.

### Why this is first
Nothing else matters if the kitchen never sees the order. Customer flow already works;
this closes the loop POS ‚Üí kitchen screen.

### Flow
```
Customer confirms on WhatsApp
  ‚Üí ordering service sets status = confirmed
  ‚Üí enqueue_partner_webhook("order.created", payload)
  ‚Üí POS kitchen screen shows order
```

### Steps

**Step 1.1 ‚Äî Order correlation fields**
- Migration: on `orders` table add:
  - `pos_order_id` (string, nullable) ‚Äî POS's ID after they ack
  - `pos_pushed_at` (timestamptz, nullable)
  - `pos_push_status` (pending/acked/failed)
- Index: `(restaurant_id, status, created_at)` for poll queries.

**Step 1.2 ‚Äî Hook after order confirmation**
- Find the ONE place order becomes `confirmed` (conversation engine checkout path
  or `ordering/service.py` confirm function).
- After `fsm_transition ‚Üí confirmed` + commit:
  - Call `enqueue_partner_webhook("order.created", order_payload)`
  - Idempotency key: `pos-order-created-{order.id}`
- **Do NOT** call POS HTTP inline in the request ‚Äî always via webhook queue.

**Step 1.3 ‚Äî Build order payload serializer**
- `partner/orders_api.py ‚Üí build_order_payload(order) ‚Üí dict`
- Include: `order_id`, `order_number`, `pos_store_id`, customer name/phone,
  items[], address, subtotal, delivery_fee, total, payment="COD",
  `promised_eta`, `sla_deadline`, `created_at`, `additional_details`.

**Step 1.4 ‚Äî Poll endpoint (backup if POS can't receive webhooks)**
- `GET /api/v1/partner/orders`
  - Query: `?status=confirmed&since=<iso>&limit=50`
  - Returns orders where `pos_push_status = pending` OR `status = confirmed` and not acked.
  - Auth: `X-API-Key` (already wired via `current_restaurant_via_api_key`).

**Step 1.5 ‚Äî Ack endpoint**
- `POST /api/v1/partner/orders/{order_id}/ack`
  - Body: `{ "pos_order_id": "POS-12345" }`
  - Sets `pos_order_id`, `pos_push_status = acked`.
  - Returns full order payload.

**Step 1.6 ‚Äî Tests**
- WhatsApp test order ‚Üí confirmed ‚Üí webhook row created with correct payload.
- Poll returns the order before ack; after ack, poll excludes it.
- Idempotency: confirming same order doesn't create duplicate webhook rows.

**Phase 1 done when:** Test order from WhatsApp appears in POS (webhook or poll) and POS can ack.

## Phase 2 ‚Äî Kitchen status IN from POS (OPS engine, ~1 week)
**Goal:** POS marks preparing/ready ‚Üí our system advances FSM ‚Üí `ready` triggers dispatch.

### Why this is the main engine
This is the critical path:
```
POS taps "Ready"
  ‚Üí POST /partner/orders/{id}/status  { "status": "ready" }
  ‚Üí advance_kitchen_status(actor="pos")
  ‚Üí _auto_dispatch_on_ready()
  ‚Üí rider app push notification
  ‚Üí customer WhatsApp "on the way"
```

### Steps

**Step 2.1 ‚Äî Status mapping table**
- `partner/status_map.py` ‚Äî dict, configurable per partner later:
  ```python
  POS_TO_OUR = {
    "accepted":  "confirmed",    # no-op if already confirmed
    "preparing": "preparing",
    "ready":     "ready",
    "cancelled": "cancelled",
  }
  ```
- Start hardcoded. Move to DB only if second POS partner needs different labels.

**Step 2.2 ‚Äî Partner status endpoint**
- `POST /api/v1/partner/orders/{order_id}/status`
  - Body: `{ "status": "preparing" | "ready" | "cancelled", "reason": "..." }`
  - Auth: `X-API-Key`
  - Logic:
    1. Load order (tenant-scoped).
    2. Map POS status ‚Üí our FSM target.
    3. If `preparing` or `ready`: call `advance_kitchen_status(session, order, actor="pos")`
       ‚Äî may need to call TWICE if jumping confirmed‚Üíready (see Step 2.3).
    4. If `cancelled`: call `cancel_order(session, order, actor="pos", reason=...)`.
    5. Audit log with `actor="pos"`.
    6. Return `{ "order_id", "status", "rider_assigned": bool }`.

**Step 2.3 ‚Äî Multi-hop advance (confirmed ‚Üí ready in one call)**
- POS may send `ready` directly without sending `preparing` first.
- Helper `advance_kitchen_to(session, order, target_status, actor)`:
  - Loop `advance_kitchen_status()` until order.status == target or illegal transition.
  - Stops at `ready` ‚Üí dispatch already fires automatically (existing code).

**Step 2.4 ‚Äî Guard: only POS-advanceable states**
- Reject if order is `draft`, `delivered`, etc. ‚Üí 422 with clear message.
- Reject if order not yet acked by POS (optional strict mode, configurable).

**Step 2.5 ‚Äî Tests (most important tests in whole project)**
- confirmed ‚Üí preparing: customer gets preparing ping, NO dispatch.
- preparing ‚Üí ready: dispatch runs, rider push sent, assignment created.
- confirmed ‚Üí ready (single call): both transitions happen, dispatch runs once.
- cancelled from preparing: order cancelled, customer notified.
- Dashboard `POST /orders/{id}/advance` still works (regression).

**Phase 2 done when:** POS "Ready" button assigns a rider and customer gets tracking message.

**Implemented once (local, revert OK):** `partner/status_map.py`, `advance_kitchen_to` +
`apply_partner_kitchen_status` in `orders_api.py`, `POST /api/v1/partner/orders/{id}/status`,
tests in `tests/partner/test_partner_kitchen.py`.
